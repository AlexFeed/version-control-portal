# ⚙️ Backend — Version Control Portal

Серверная часть системы контроля версий и конфигураций. Разработана на **NestJS**, использует **Prisma ORM** и **PostgreSQL** (через Docker). Обеспечивает транзакционное управление данными, ролевой доступ по JWT-токенам, безопасное хеширование паролей и загрузку медиа-файлов (аватары).

---

## 🚀 Как развернуть проект

Процесс установки сведён к максимально простому процессу благодаря внедрению DOCKER.

### Предварительные требования
1. Необходимо установить Node.js, а также docker desktop
2. Запуситить docker desktop в фоне и не выходить из него
3. Создать файл .env в корне папки /backend для конфигруации (на этапе разработки и тестирования можно просто скопировать текст из .env.example и вставить в новый созданный .env)
4. Откройте терминал в директории `backend` (cd backend) и выполните команды:
  ```bash
  npm install # Установка библиотек
  npm run backend:dev # Ниже написано что делает эта команда 
  ```

### Что скрывается под капотом этой команды?
Команда `npm run backend:dev` последовательно выполняет следующие шаги:
1. `docker-compose up -d` — поднимает контейнер с чистой PostgreSQL в фоновом режиме.
2. `npx prisma generate` — генерирует типы Prisma Client для TypeScript.
3. `npx prisma migrate dev` — создает таблицы и настраивает связи в базе данных.
4. `npx prisma db seed` — заполняет базу начальными данными (создает тестовые проекты и пользователей).
   > **Важно:** Сид автоматически создает базового администратора для входа в систему. Пароли в базе хранятся в **захешированном виде** (через bcrypt).  
   > **Логин:** `admin`  
   > **Пароль:** `admin123`
5. `nest start --watch` — запускает сам сервер на `http://localhost:3000` в режиме горячей перезагрузки (Hot-Reload).

## 🗄 Структура Базы Данных

База данных состоит из 5 основных сущностей и связующих таблих. Prisma обеспечивает строгое каскадное удаление.

*   **User (Пользователи):** Хранит учетные данные (`login`, захешированный `password`, `avatarUrl`). Имеет поле `role` (Enum: `ADMIN`, `DEVELOPER`, `VIEWER`). Связан с таблицами `Version` и `MemberProject`.
*   **Project (Проекты):** Хранит `name` и `description`. Основной контейнер для релизов. Имеет связи с участниками (`MemberProject`).
*   **Version (Версии):** Хранит номер `version` (строка, например "1.0.0") и описание `description`. Содержит внешние ключи `projectId` и `authorId`.
*   **Change (Изменения):** Хранит текст изменения `description`. Содержит внешний ключ `versionId`. Полностью управляется через контроллер версий.
*   **MemberProject (Участники проектов):** Связующая таблица (Many-to-Many) между пользователями и проектами. Определяет, в каких проектах участвует конкретный пользователь.

## 🔌 API Эндпоинты и Форматы Данных

Все запросы (кроме `/auth/login` и раздачи статики `/uploads`) защищены и требуют передачи заголовка: 
`Authorization: Bearer <ваш_токен>`

Базовый URL: `http://localhost:3000`

### 1. Авторизация (Auth)

#### Вход в систему
*   **Метод:** `POST /auth/login`
*   **Доступ:** Открытый (без токена)
*   **Входные данные (Body):**
    ```json
    {
      "login": "admin",
      "password": "admin123"
    }
    ```
*   **Выходные данные (200 OK):** Возвращает JWT токен и объект пользователя.
    ```json
    {
      "access_token": "eyJhbGciOiJIUzI1NiIsInR5c...",
      "user": {
        "id": 1,
        "login": "admin",
        "role": "ADMIN",
        "avatarUrl": "/uploads/avatars/admin.png"
      }
    }
    ```

---

### 2. Пользователи (Users)

#### Получение списка пользователей
*   **Метод:** `GET /users`
*   **Доступ:** `ADMIN`, `DEVELOPER`, `VIEWER`
*   **Выходные данные (200 OK):** Массив объектов пользователей (пароли строго исключены из выборки на уровне сервиса).
    ```json
    [
      { "id": 1, "login": "admin", "role": "ADMIN", "avatarUrl": null }
    ]
    ```

#### Создание пользователя
*   **Метод:** `POST /users`
*   **Доступ:** Только `ADMIN`
*   **Входные данные (Body):** Поле `role` должно быть одним из: `ADMIN`, `DEVELOPER`, `VIEWER`. Пароль будет автоматически захеширован.
    ```json
    {
      "login": "new_user",
      "password": "secret_password",
      "role": "DEVELOPER"
    }
    ```

#### Загрузка аватара пользователя
*   **Метод:** `POST /users/:id/avatar`
*   **Доступ:** Только сам пользователь (проверка по ID из токена).
*   **Формат:** `multipart/form-data` с полем `file`. Картинка сохраняется в `/uploads/avatars/`, а в базу записывается путь к ней.

#### Получение проектов пользователя
*   **Метод:** `GET /users/:id/projects`
*   **Доступ:** Авторизованные пользователи.
*   **Выходные данные (200 OK):** Возвращает массив проектов, в которых состоит пользователь. В выборку попадают как проекты, назначенные через таблицу `MemberProject`, так и те, в которых пользователь является автором хотя бы одной версии. Слияние происходит с фильтрацией дубликатов на уровне сервиса.

---

### 3. Проекты (Projects)

#### Чтение проектов
*   **Получить все:** `GET /projects` (Доступ: все роли). Возвращает массив проектов.
*   **Получить один:** `GET /projects/:id` (Доступ: все роли). Возвращает конкретный проект по ID.

#### Создание проекта
*   **Метод:** `POST /projects`
*   **Доступ:** `ADMIN`, `DEVELOPER`
*   **Входные данные (Body):**
    ```json
    {
      "name": "E-commerce Platform",
      "description": "Основной проект компании"
    }
    ```

#### Обновление проекта
*   **Метод:** `PATCH /projects/:id`
*   **Доступ:** `ADMIN`, `DEVELOPER`

#### Удаление проекта
*   **Метод:** `DELETE /projects/:id`
*   **Доступ:** `ADMIN`, `DEVELOPER`
*   **Поведение:** Каскадно удаляет все `Versions` и `Changes`, привязанные к этому проекту.

#### Работа с участниками проекта
*   **Метод GET:** `GET /projects/:id/members`
    *   Возвращает объединенный список участников проекта. Сюда входят как пользователи, назначенные вручную (через `MemberProject`), так и авторы всех загруженных версий.
*   **Метод PUT:** `PUT /projects/:id/members`
    *   Доступ: `ADMIN`
    *   Принимает массив `userIds`. Внутри транзакции удаляет все текущие ручные связи для проекта и создает новые записи, обеспечивая точное соответствие базы переданному массиву.
*   **Методы POST/DELETE:** `/projects/:projectId/members/:userId` для точечного добавления и удаления связей.

### 4. Версии и Изменения (Versions & Changes)

*Особенность архитектуры: Изменения (`changes`) управляются контроллером Версий. При создании и обновлении бэкенд ожидает плоский массив строк, что идеально совместимо с фронтендом.*

#### Чтение версий
*   **Получить все:** `GET /versions` (Доступ: все роли). Массив изменений (`changes`) подтягивается автоматически.
*   **Получить одну:** `GET /versions/:id` (Доступ: все роли).

#### Создание версии с изменениями
*   **Метод:** `POST /versions`
*   **Доступ:** `ADMIN`, `DEVELOPER`
*   **Входные данные (Body):**
    *   `projectId`: ID существующего проекта.
    *   `changes`: Массив строк (должен содержать минимум 1 элемент).
    > ID автора (`authorId`) передавать не нужно — бэкенд безопасно извлечет его из переданного JWT-токена.

#### Обновление версии (и каскадное обновление изменений)
*   **Метод:** `PATCH /versions/:id`
*   **Доступ:** `ADMIN`, `DEVELOPER`
*   **Входные данные (Body):** Если в теле запроса передается массив `changes`, **все старые изменения этой версии удаляются из базы данных, а переданные записываются как новые**. Это избавляет фронтенд от необходимости высчитывать ID добавленных и удаленных пунктов в интерфейсе.

#### Удаление версии
*   **Метод:** `DELETE /versions/:id`
*   **Доступ:** `ADMIN`, `DEVELOPER`
*   **Поведение:** Каскадно удаляет все привязанные строки из таблицы `Change`.
